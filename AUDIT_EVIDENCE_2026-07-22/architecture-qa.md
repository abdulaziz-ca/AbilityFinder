# AbilityFinder audit — Passes 1–3 (architecture, automated checks, state space)

Audit date: 2026-07-22 (America/Edmonton). Repository remained unmodified; `git status --short` and `git diff --check` were clean after testing. Evidence scripts are in `/tmp/abilityfinder-audit/` only.

## Scope and architecture map

- Deployment: one Cloudflare Worker (`src/index.js`) with `public/` as the sole asset directory (`wrangler.jsonc:3-13`). Static fallthrough is `env.ASSETS.fetch()` (`src/index.js:349-357`). API routes are `POST/OPTIONS /api/ask`, `POST/OPTIONS /api/feedback`, and public `GET /api/link-health`; a three-hour scheduled link checker writes aggregate official-link health to KV (`src/index.js:349-387`, `wrangler.jsonc:24-34`).
- Browser application: classic scripts, no build. `public/index.html` loads persistence and theme first, then catalog/directory files and `public/app.js`. Startup awaits migration/restore before the first meaningful render (`public/app.js:915-968,4445-4453`).
- Internal views (history state, not path-addressable URLs): `landing`, `wizard`, `results`, `browse`, `detail`, `privacy`, `about`, `support`, `updates`, `help`, `accessibility`, `professionals`, `partner-overview`, `impact`, `dtc-prep`, `grants`, `organizations` (`public/stateManager.js:38`; router `public/app.js:1255-1339`). Parameterized surfaces add 84 benefit detail IDs and 8 help topics. Static routes/assets additionally include `/embed(.html)`, `/guides/`, 83 generated individual guide pages, `/404.html`, `/robots.txt`, and `/sitemap.xml`.
- Questionnaire: 17 declared steps, 10–17 visible depending on province, selected disability, and city availability (`public/app.js:550-872`). Max path is BC + autism/intellectual + physical (all three BC-only questions, autism diagnosis/onset, mobility, and city). The 12 disability categories are multi-select; eight age bands drive functional/situation options; AB has 51 city choices, BC 45, and `other` skips city.
- Eligibility: 97 requirement predicates are defined, 86 used by 84 benefits. `evaluate()` returns `no` when any unmet requirement is fixed, `almost` when only actionable requirements remain, and `ready` otherwise (`public/app.js:525-548`). Results suppress `no` entries belonging to another province (`public/app.js:2739-2748`).
- Catalog/supports: 84 benefit records, 19 supports, 12 help organizations, separate grants and organization directories. Static invariants found no duplicate benefit IDs, unknown requirement names, orphan metadata IDs, missing required benefit/detail fields, or municipalities declared without a corresponding local benefit. `local-supports` is the sole benefit without a generated static guide (83 guides for 84 benefits); its URL is answer-dependent and it functions as a generic fallback.
- Persistence: `stateManager.js` allowlists 19 answer properties plus route/UI state and catalog-valid IDs (`public/stateManager.js:15-40,77-183`). `dbManager.js` owns IndexedDB, serializes snapshots, queues same-tab writes, uses optimistic revisions for cross-tab conflicts, and writes metadata-only tombstones on clear (`public/dbManager.js:37-167,227-275`). Legacy localStorage is sanitized before migration and only removed after a successful write or an authoritative IDB record/tombstone (`public/dbManager.js:169-225`).
- Generated assets: `src/benefits-context.js` and `src/links.js` are generated from benefit/directory inputs by `scripts/gen-benefits-context.js`; static guides/sitemap are generated by `scripts/gen-guide-pages.js`. They were inspected as generated outputs, not treated as hand-maintained sources.

## Questionnaire coverage ledger

The exact count below is for **completed semantic answer sets reachable through a fresh current UI**, treating selection order within a multi-select as irrelevant. It excludes action-history permutations, partial journeys, malformed/legacy states, and persisted UI preferences.

- Disability selections plus dynamically visible autism/onset/mobility answers: 23,039 combinations.
- Age-dependent functional-needs × situation combinations across all eight age bands: 178,270.
- Province/city and BC-only MSP/assistance/circumstance branches: 4,912.
- Persona × documentation × citizenship × DTC × income: 162.
- Exact reachable completed semantic states: **3,268,249,940,272,320**.

Full Cartesian execution is infeasible. A deterministic reduced-equivalence run executed **250,000 unique complete states**. It covered all 3 personas, 8 age bands, 3 residency values, every 51 AB and 45 BC cities plus the out-of-scope null-city state (97 province/city classes), every documentation/DTC/income value, 64 combinations of the six disability values that influence benefit predicates, all 392 age-valid functional-need projections, all 94 age-valid situation projections, and all 10 circumstance projections. It observed **186,872 distinct 84-benefit status vectors**, 2,328,858 `ready` evaluations, 1,586,046 `almost`, 17,085,096 `no`, no evaluator exceptions, 11 one-match states, 249,989 many-match states, and no zero-match state.

This is broad combinatorial coverage, not exhaustive raw-state coverage. Partial/back-navigation histories, every order of multi-selection, persistence failures, and DOM rendering were not included in that 250,000-state evaluator run. Existing E2E tests separately cover one full AB journey, several AB/BC adult/child paths, mid-flow reload, legacy migration, one stale-tab conflict, unavailable IDB, and 3 personas × 17 persisted views (51 view/persona reloads).

Zero results are structurally unreachable: the DTC benefit requires `prolonged` and `certifier` (`public/data.js:917-926`), while both predicates always return true (`public/app.js:202-203`). The zero-result branch in `resultsBlurb()` (`public/app.js:2941-2943`) is therefore dead under current data. This may be an intentional universal entry point, but it means a requested zero-results journey cannot be exercised without changing fixtures/source and the UI never displays its authored recovery copy.

## Findings

### AQ-01 — Analytics implementation conflicts with the repository’s non-negotiable privacy boundary

- Severity/Priority/Confidence/Type: **Medium / P1 / Confirmed / Privacy + documentation drift**.
- Evidence labels: **Code-confirmed**. Production beacon execution is **manual production validation required**.
- Affected users/routes: all visitors and all static pages if Cloudflare injects the beacon.
- Evidence: `AGENTS.md:32-35` says no analytics; `AGENTS.md:136-138` and `DEPLOY.md:116-119` require the strict CSP to block injected analytics. Current `public/_headers:10` explicitly permits `https://static.cloudflareinsights.com` scripts and `https://cloudflareinsights.com` connections. `public/app.js:1934-1936` says the product uses Cloudflare Web Analytics. `e2e/persistence.spec.js:3-22` allowlists those origins and aborts them rather than failing on their presence.
- Expected: the implementation and tests enforce the stated no-analytics boundary; injected analytics remain blocked.
- Actual: CSP, privacy copy, and tests deliberately support Web Analytics.
- Likely root cause: product/privacy policy changed without updating the controlling AGENTS/DEPLOY rules, or analytics was reintroduced contrary to those rules.
- Correction/verification: owner must choose and document one policy. Under current mandatory instructions, remove the analytics allowances and disclosure, disable injection at Cloudflare, make E2E fail any non-local request, then validate production response headers and network traffic. Add a contract test tying `_headers`, privacy copy, allowed request origins, and AGENTS/DEPLOY language together.

### AQ-02 — The production assistant still tells BC users the product does not cover BC

- Severity/Priority/Confidence/Type: **Medium / P1 / High confidence / Defect + data-integrity risk**.
- Evidence: **Code-confirmed**; live model behavior **needs isolated/production validation**.
- Affected: BC users of `/api/ask`, especially questions about BC programs.
- Evidence: BC is enabled (`public/app.js:9`) and the generated catalog contains BC programs, but `src/index.js:33-36` frames the tool around disabled Albertans/Canadian or Alberta programs and the final overriding instruction says “Scope is Alberta and federal Canada only. For another province, say AbilityFinder does not cover it yet” (`src/index.js:72`).
- Expected: grounded assistant scope agrees with the active AB+BC catalog.
- Actual: prompt instructions conflict; a BC question can be refused or answered inconsistently despite a matching grounded BC record.
- Root cause: BC launch updated the catalog/client/generators but not the hand-written system-prompt scope.
- Correction/verification: update the scope only after reviewing assistant safety; add mocked Worker tests for a BC program, BC follow-up, out-of-scope province, and retrieval precedence, then run the documented regression prompts against the deployed model without personal data.

### AQ-03 — British Columbia browse filter is silently lost on save/reload

- Severity/Priority/Confidence/Type: **Low / P2 / Confirmed / Defect (persistence/UX)**.
- Evidence: **Observed** in a direct `buildPersistedState()` call and **Code-confirmed**.
- Reproduction: set `browseLevel = "British Columbia"`, build/persist state, reload. `ui.browseLevel` becomes `"all"`.
- Evidence: app exposes `British Columbia` as a valid browse level (`public/app.js:3652-3657`), but the persistence allowlist omits it (`public/stateManager.js:39`) and falls back to `all` on build/restore (`public/stateManager.js:142,174`). The E2E persistence test uses only `Federal`, while the BC filter test never reloads.
- Expected: the active BC filter restores just like Federal/Alberta/local.
- Actual: reload expands the list to all 84 benefits, increasing cognitive load and making the saved UI state inaccurate.
- Correction/verification: add the current browse-level key to the state boundary and a catalog-backed invariant; E2E select BC, reload, assert the chip remains active and non-BC cards stay absent.

### AQ-04 — Persistence accepts cross-province city combinations and can suppress local fallback help

- Severity/Priority/Confidence/Type: **Medium / P2 / High confidence / Reliability defect**.
- Evidence: **Observed** sanitizer output and **Code-confirmed** consequence.
- Affected: users restoring malformed, manually altered, or legacy state; local program matching.
- Reproduction: sanitize/build a state with `{province:"AB", city:"Vancouver"}` using the current global city allowlist. The pair survives. On evaluation, `REQS.cityOther` is false because Vancouver is in `CITIES_WITH_PROGRAMS` (`public/app.js:398-404`), BC city benefits fail `bc`, and no Alberta-specific city predicate matches, so local fallback help is omitted.
- Evidence/root cause: city validation uses one combined catalog (`public/app.js:874-887`); `cleanAnswers()` validates each property independently and only clears city for provinces other than AB/BC (`public/stateManager.js:77-94`). Current UI selection normally clears the pair (`public/app.js:704-706`), but the persistence boundary claims to protect malformed/legacy state.
- Expected: restored city belongs to the restored province or is cleared so the user must choose again.
- Actual: a catalog-valid but jurisdiction-invalid pair survives and silently changes results.
- Correction/verification: validate city against province-specific selections, then add unit cases for AB+BC city, BC+AB city, unknown city, `other`+city, and an E2E malformed-snapshot recovery that resumes at city rather than presenting incomplete local results.

### AQ-05 — Eligibility and result-state logic has no systematic automated regression suite

- Severity/Priority/Confidence/Type: **Medium / P1 / Confirmed / Test gap**.
- Evidence: **Code-confirmed** test inventory and the coverage ledger above.
- Affected: all users; any catalog/REQS change can silently alter high-stakes results.
- Evidence: 29 unit tests cover persistence/privacy contracts, not `REQS`, `evaluate`, result grouping, benefit/city invariants, or outcome snapshots. The E2E suite asserts selected programs for a small set of AB/BC journeys. Current semantic completed space is 3.268 quadrillion; the audit’s 250,000-state run alone produced 186,872 distinct status vectors.
- Expected: data-driven invariant tests plus boundary/equivalence cases for every benefit requirement and jurisdiction.
- Actual: a requirement typo is partly protected only by runtime behavior/static manual inspection; result regressions outside the named examples have no automated oracle.
- Correction/verification: add a read-only exported/test harness for the eligibility engine; assert every requirement resolves, every benefit attains its intended statuses under documented fixtures, cross-jurisdiction exclusion, city fallbacks, age boundaries, uncertainty semantics, determinism, and canonical snapshots for representative P0/P1 journeys. Generate a coverage ledger in CI without claiming raw exhaustiveness.

### AQ-06 — Browser suite validates a static Chromium simulation, not the deployed Worker/security behavior or browser matrix

- Severity/Priority/Confidence/Type: **Medium / P2 / Confirmed / Test gap**.
- Evidence: **Code-confirmed**.
- Evidence: `playwright.config.js:3-16` declares no browser projects and serves `public/` through Python. Therefore the 21 passing tests are Chromium-only and cannot exercise Worker API routing, `_headers`/CSP, Cloudflare asset 404 handling, real rate-limit/binding failures, assistant stream behavior, feedback, link health, or scheduled behavior. There are no Worker tests. Fixed `waitForTimeout(230)` calls (`e2e/bc-live.spec.js:22-30`) couple journeys to implementation timing and contributed to a 2.1-minute suite.
- Expected: fast static UI tests plus a focused Wrangler/miniflare/Worker layer and Chromium/Firefox/WebKit critical-path matrix.
- Actual: green E2E can coexist with API/header/production-scope defects such as AQ-01/AQ-02.
- Correction/verification: retain static tests for speed; add Worker contract tests with fake bindings/streams; add a small cross-browser critical suite; use state/DOM waits instead of fixed sleeps; test history back/forward, reset/tombstone behavior, corrupted/delayed/rejected IDB, route render exceptions, zero/one/many-result fixtures, and asset/API failure modes.

### AQ-07 — Current handoff describes pre-BC scope and an incomplete route model

- Severity/Priority/Confidence/Type: **Informational / P3 / Confirmed / Maintainability**.
- Evidence: **Code-confirmed** documentation drift.
- Evidence: `HANDOFF.md:18-20` says other jurisdictions are parked/not loaded, while `public/app.js:9`, `public/data.js:70-84`, 48 BC benefit records, BC E2E tests, and generated BC guides show BC live. The handoff’s view summary omits newer `accessibility`, `professionals`, `partner-overview`, `impact`, `dtc-prep`, `grants`, and `organizations` views even though its later file table mentions some features.
- Impact: future audits and changes can select the wrong scope; this already appears to have contributed to AQ-02.
- Correction/verification: update current-state docs after implementation corrections; add a lightweight contract comparing active coverage/view constants with critical scope documentation or generate current counts instead of hand-maintaining them.

## Test-suite reliability and missing cases

Strengths: state allowlisting, legacy privacy, queued writes, stale-save-after-clear, stale-clear-after-save, all 17 persisted view types for all 3 personas, mid-flow restore, unavailable IndexedDB, age-band migration, AB/BC jurisdiction examples, and 320px overflow are asserted and passed. Assertions are generally observable rather than vacuous.

Material gaps: no eligibility-engine unit tests; no Worker/API/stream/CSP tests; no Firefox/WebKit; no axe integration (and automated accessibility would still be insufficient); no actual screen-reader/keyboard full journey; no direct history back/forward test; no reset/clear/resurrection browser test; no IDB corruption, delayed open, quota/rejected transaction, or blocked upgrade; no offline/failed asset/API journey; no full municipality matrix; no zero-result executable path; no print/calendar tests in the current suites; no route-injected render exception test; no generated-output freshness/diff check; no current online dependency advisory result. The suite also deliberately ignores/aborts Cloudflare analytics, so it cannot demonstrate the stricter documented no-analytics boundary.

## Pass 3 limitations and follow-up regression design

The 250,000-state run executes the actual current predicates/data in a VM and is deterministic, but it does not render DOM, persist each model, or establish that each observed output is factually correct. It collapses disability labels that no benefit predicate reads and treats multi-select order as irrelevant. Every discovered code defect should receive a small deterministic regression rather than adding more random samples. For P1/P2: BC assistant scope contract; BC browse reload; cross-province city sanitization and result fallback; requirement/reference invariants; age boundaries; each fixed/actionable/uncertain predicate; AB/BC/other jurisdiction isolation; zero/one/many fixture policy; Worker headers/API failures; and cross-browser critical journeys.
